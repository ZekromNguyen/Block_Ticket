# BlockTicket AWS Infrastructure

This Terraform/OpenTofu stack provisions a production-oriented AWS baseline for the portfolio deployment:

- VPC with public/private subnets across multiple availability zones
- EKS managed control plane and node group
- Immutable ECR repositories for each service image
- RDS PostgreSQL for managed relational storage
- ElastiCache Redis for cache/locking workloads
- Amazon MQ RabbitMQ for asynchronous messaging
- AWS Secrets Manager for application secrets
- IRSA role for External Secrets Operator
- **Shared S3 state bucket + DynamoDB lock table** (bootstrap)

## Directory layout

```
infra/terraform/aws/
├── bootstrap/                 # One-time: provisions the S3 state bucket + DynamoDB lock table.
├── modules/
│   ├── platform/              # VPC, EKS, RDS, ElastiCache, AmazonMQ, ECR, Secrets Manager.
│   └── external-secrets-irsa/ # IAM role granting the ESO service account access to Secrets Manager.
├── environments/
│   ├── dev/                   # dev env
│   ├── staging/               # staging env (promoted from dev via `v*.*.*-rc*` tags)
│   └── prod/                  # prod env (promoted from staging via `v*.*.*` tags)
└── scripts/
    ├── init-backend.sh        # Linux/macOS bootstrap helper
    └── init-backend.ps1       # PowerShell variant
```

## Apply order (from a clean machine)

```bash
# 1. One-time: provision the shared state backend.
cd infra/terraform/aws/bootstrap
terraform init
terraform apply

# 2. Per environment: initialise against the bucket the bootstrap step created,
#    then validate/plan/apply. The script does the init wiring and then you run
#    the normal plan/apply cycle.
cd ../environments/dev
../../scripts/init-backend.sh dev
terraform plan -out=tfplan-dev
terraform apply tfplan-dev

# Promote:
cd ../staging && ../../scripts/init-backend.sh staging && terraform apply
cd ../prod    && ../../scripts/init-backend.sh prod    && terraform apply
```

PowerShell equivalent uses `init-backend.ps1`.

After Terraform apply, configure kubectl:

```bash
aws eks update-kubeconfig --region ap-southeast-1 --name blockticket-<env>
```

Then install External Secrets Operator into the cluster and apply `k8s/addons/external-secrets/`.

## Generated secrets

The Secrets Manager value auto-generated by the platform module contains placeholders for JWT, encryption, SMTP, and Twilio secrets. Rotate those values before exposing the application. See `infra/terraform/aws/modules/platform/main.tf` lines 420–435 for the exact keys.

## Static analysis caveats

`checkov` will warn about a few settings that are intentional — see `.checkov.yaml` at the repo root for soft-skip documentation. In brief:

- S3 bucket uses AES256 (fine for now; switch to KMS for compliance workloads)
- S3 server-access logging is stubbed out until a destination bucket is provisioned
- IAM auth for RDS is not wired yet — services use Secrets Manager credentials
